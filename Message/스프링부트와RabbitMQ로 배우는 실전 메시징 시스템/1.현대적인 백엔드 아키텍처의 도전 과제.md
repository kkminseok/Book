- 왜 RabbitMQ 같은 메시징 시스템을 배워야하는가?
    - 모놀리식 아키텍처의 한계
    - 장점
        - 개발 단순성
        - 쉬운 배포
        - 통합 테스트의 용이성
        - 성능 (초기에는)
    - 단점
        - 거대 진흙 덩어리
        - 코드의 복잡성과 강한 결합
        - 스파게티 코드
        - 이해의 어려움
        - 수정의 두려움
        - 기술 스택의 경직성
        - 덩치가 커질수록 느린 개발 및 배포 속도
        - 확장성의 한계
        - SPOF

- MSA와 통신의 복잡성
    - 특징
        - 독립적 코드베이스
        - 독립적 데이터베이스
        - 독립적 배포
        - 기술 다양성
    - 장점
        - 결합도 감소
        - 유연한 기술 선택
        - 빠른 배포
        - 선택적 확장
        - 장애 격리
    - 문제
        - 통신의 복잡성
        - 어떻게 통신할 것인가(프로토콜)
        - 서로 어떻게 찾을 것인가(Service Discovery)
        - 어떻게 신뢰할 수 있는 통신을 할 것인가(Reliability)
        - 데이터 일관성은 어떻게 유지할 것인가(Data Consistency)
- 동기(Synchronous) 통신의 함정과 장애 전파
    - MSA에서 동기통신을 통해 여러 서비스에 접근하면 서비스를 시간적으로 묶어버려 장점을 활용할 수 없다.
    - 문제
        - 가용성 종속(하나 잘못되면 장애로 이어질 수 있음.)
        - 성능 종속(전체 응답시간은 전체 처리시간 합산)
        - 장애 전파(타임아웃으로 인한 쓰레드풀 고갈)


위 문제를 해결하기 위해서 **비동기 메시징**이 필요하게됨. 이 아이디어를 구현해주는 시스템이 **메시지 큐**이다.
